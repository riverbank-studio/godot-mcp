# Docs / retrieval / SQLite review

Scope: docs ingestion + retrieval (DESIGN.md §§ Documentation subsystem, Distribution → Bundled docs DB / Embedding model) and the docs-flavored research items (#38, #39). Tool routing/descriptions, latch semantics, LSP, npm size, and benchmark dataset design are punted to their owners (see Out of scope).

## Findings

- **[severity: high]** **[scope: design]**
  DESIGN.md § Ingestion pipeline step 2 says "Fetch godot tarball ... Sparse extract: only `doc/classes/`." `codeload.github.com` serves `git archive`-style tarballs in a single sequential stream and the documented archive endpoints do not support path filtering or partial downloads (GitHub docs only describe "complete snapshots" — `git archive` tarball or zipball). "Sparse extract" therefore can only mean "stream the full tarball but retain only entries matching `doc/classes/`," and DESIGN.md should say so explicitly so an implementer doesn't waste a day looking for a `?path=` parameter. The alternative is the GitHub Git Trees API with `?recursive=1` (limit 100,000 entries / 7 MB per response, which `godotengine/godot`'s `doc/classes/` directory is well under) followed by per-file Blob fetches — but that's many more requests and rate-limit-sensitive. Recommend: keep tarball, rename to "stream-extract" and document the discarded-bytes cost in DESIGN.md, OR switch to trees+blobs API with explicit ETag/conditional-request notes.
  *Recommended action:* edit DESIGN.md § Ingestion pipeline step 2.

- **[severity: high]** **[scope: design]**
  Tutorial chunk hard cap is **3000 tokens** but `sentence-transformers/all-MiniLM-L6-v2` has a max sequence length of **256 word-pieces** (see https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2 — not 512 as my prompt assumed; it's even tighter). Any chunk over ~200 tokens of real text will be silently truncated by the tokenizer, meaning the embedding only "sees" the first ~256 tokens of a 3000-token chunk and the remaining 90%+ contributes nothing to dense retrieval. The hard cap currently binds storage size, not embedding semantics. Either (a) tighten the hard cap toward the model's window (e.g. soft 200 / hard 256 word-pieces, accept smaller chunks), (b) pick a model with a larger context window (BGE-small / GTE-small are 512, mxbai-embed-xsmall is 4096), or (c) embed a "summary view" (title + heading_path + first N tokens) rather than the whole chunk and accept that lexical FTS5 carries the long tail. This is the single most consequential docs decision — pick one before benchmark #32 runs.
  *Recommended action:* edit DESIGN.md § Ingestion pipeline step 6/7 and § Search → Tutorials, then revisit #32 acceptance criteria.

- **[severity: medium]** **[scope: design]**
  MiniLM-L6-v2 is from 2021 (Apache 2.0, 22.7M params, 384-dim, MTEB Retrieval ~41 on the leaderboard at time of release). Current ~33M-param contenders meaningfully outperform it on MTEB retrieval: **BAAI/bge-small-en-v1.5** (Sept 2023, 33.4M params, 384-dim, **512 ctx, MTEB retrieval 51.68**, MIT) and **thenlper/gte-small** (Aug 2023, 33.4M params, 384-dim, 512 ctx, MTEB retrieval 49.46, MIT). Both are drop-in compatible with `@xenova/transformers` (ONNX export), share MiniLM's 384-dim output (so sqlite-vec schema is unchanged), have permissive licenses, and roughly double the context window. BGE-small is the strongest small open model for retrieval as of the design's writing. Either justify MiniLM-L6 explicitly (e.g. "we pick the smallest 22M-param option to minimize bundle"), or switch — there is no neutral case for shipping a 2021 model with worse retrieval and a tighter context window when a 10MB-larger replacement is strictly better. (mxbai-embed-xsmall-v1 is the same family as MiniLM but with Matryoshka + AnglE retraining; another reasonable upgrade.)
  *Recommended action:* edit DESIGN.md § Ingestion pipeline step 7 and § Embedding model — pick BGE-small-en-v1.5 or document the MiniLM trade-off.

- **[severity: medium]** **[scope: design]**
  DESIGN.md § Embedding model: "Bundled in the package (~80MB). Total npm package size approximately 150MB." `@xenova/transformers` already supports auto-download + on-disk cache on first use (default cache `/models/`; configurable via `localModelPath`; can be disabled with the `allowRemoteModels` flag). Shipping the ONNX weights inside the npm tarball is a real DX cost: every `npm install` pays the bandwidth, including CI runs that won't touch tutorials. Three options worth considering: (a) bundle, current plan, ~150 MB tarball; (b) auto-download on first tutorial search, no bundle, ~70 MB tarball, ~1 s + ~80 MB network on first call after install — same UX cost as the bundled DB approach already in DESIGN.md for `latest`; (c) "lazy install" via a `postinstall` script that downloads after `npm install` finishes. Option (b) is the closest match to DESIGN.md's existing posture ("fast startup on common path"). Trade-off is a network dependency for first-time tutorial use and a small supply-chain attack surface (HF Hub becomes a trust anchor) — fixable by pinning the model SHA and verifying on download. Flag this as a deliberate decision in DESIGN.md either way.
  *Recommended action:* edit DESIGN.md § Embedding model with rationale; consider opening a new issue for "evaluate lazy-download embedding model" if you want the option.

- **[severity: medium]** **[scope: design, issue:#38]**
  Drizzle ORM is **not** at 1.0 stable as of the design's writing. The releases page shows latest stable is **0.45.2** (March 2025) and **v1.0.0-rc.3** (May 2025) is the latest pre-release. The "Currently, there are no SQLite extensions natively supported by Drizzle. Once those are added, we will have them here!" note is still up on `orm.drizzle.team/docs/extensions/sqlite`. So both the version and the FTS5/sqlite-vec stories that DESIGN.md hand-waves to ("ORM is for ergonomics on the normal tables; FTS5 queries and sqlite-vec calls go through raw SQL with bound parameters") are real: virtual tables and extension loading remain unsupported by Drizzle's query builder and you'll be writing raw SQL via `db.run(sql\`...\`)` for everything that touches FTS5 or vec0. That's workable, but it means Drizzle's *only* role here is type-safe access to the normal tables (classes, members, meta). Given that, the fallback path ("if Drizzle 1.0 slips ... use better-sqlite3 directly with hand-written prepared statements") is small enough that I'd recommend just doing that now and not blocking on Drizzle 1.0 — the ergonomics savings on five normal tables are modest. At minimum, set a tripwire date in DESIGN.md ("if 1.0 not GA by YYYY-MM-DD, default to better-sqlite3 direct").
  *Recommended action:* edit DESIGN.md § ORM choice with a concrete tripwire date, and update #38 with the current 1.0-rc.3 / 0.45.2 status above.

- **[severity: medium]** **[scope: design, issue:#39]**
  DESIGN.md § Search → Class reference says "BM25 weighting weights name fields ~3x description" but never gives the exact `bm25(table, weights...)` call. SQLite's FTS5 `bm25()` is fully positional: missing trailing weights default to 1.0; extras are ignored. With a single `members` FTS5 table over (name, signature, description) the call is roughly `bm25(members_fts, 3.0, 1.5, 1.0)` — but signature isn't mentioned in DESIGN.md and the classes table has (name, brief). Either commit to specific numeric weights in DESIGN.md (so the benchmark has something to A/B against) or punt explicitly to #39 with a "default of `bm25(table, 3.0, 1.0)` for two-column tables and `bm25(table, 3.0, 2.0, 1.0)` for three-column" as a starting point. Right now #39 says "verify or adjust" without baseline numbers — that makes the benchmark harder to interpret. Also: DESIGN.md doesn't mention the FTS5 tokenizer (default `unicode61` is fine for prose but `trigram` is materially better for partial class-name lookups like `Anim` → `AnimationPlayer`); call this out in #39 as well.
  *Recommended action:* edit DESIGN.md § Search → Class reference with explicit baseline weights; expand #39 body with starting-point weights and tokenizer options.

- **[severity: medium]** **[scope: design]**
  DESIGN.md asserts "WAL mode enabled. Improves concurrent read performance for the read-only post-ingestion case." This is overstated. WAL helps when readers coexist with a writer — which by design does not happen post-ingestion (the docs DB is written once during ingest and only read thereafter). Pure read concurrency on a rollback-journal DB is already excellent in SQLite (shared locks, no writer to block). What WAL *does* buy you here is a slightly different open-mode and a small startup-cost difference; what `query_only` buys you is a hard guarantee that no tool path can accidentally mutate the bundled DB. Recommendation: open the connection with `PRAGMA query_only = 1` (per https://www.sqlite.org/pragma.html — produces `SQLITE_READONLY` on any mutation attempt) and keep WAL only if ingestion-time benchmarks show it helps during the ingest itself; otherwise drop WAL mode for the bundled-DB path to keep the on-disk file to a single `.db` file (no `-wal`/`-shm` siblings to ship in the npm tarball — these would have to be cleaned up before publish anyway).
  *Recommended action:* edit DESIGN.md § Storage with `query_only = 1` and revisit WAL claim.

- **[severity: medium]** **[scope: design]**
  Lock file at `docs-{version}.lock` is fine for cross-process mutual exclusion of *ingestion*, but `better-sqlite3` opens the underlying SQLite file with OS-level POSIX/Windows locks of its own. Two MCP instances opening the same `.db` for read are fine (shared locks). The risk is the atomic-rename pattern (`.tmp` → final): on Windows, a second process holding an open read-only handle to the destination file *can* block `rename()` (EBUSY / `ERROR_SHARING_VIOLATION`), which would surface as ingestion failure. DESIGN.md should either (a) document that the lock file must be held across the rename and that other readers should close their connection before the rename (impractical), or (b) use a different rename target on Windows (rename old destination to `.bak`, rename `.tmp` to destination, delete `.bak`), or (c) accept that concurrent ingestion + concurrent reads of the same version key are a transient failure and surface it as such. Worth one paragraph in DESIGN.md.
  *Recommended action:* edit DESIGN.md § Concurrency to address Windows file-locking under atomic rename.

- **[severity: medium]** **[scope: design]**
  RRF k=60 in DESIGN.md is the right default but the citation should be explicit. The canonical reference is **Cormack, Clarke, Büttcher (2009), "Reciprocal Rank Fusion outperforms Condorcet and individual Rank Learning Methods," SIGIR'09**, and the k=60 default is what Elastic, Vespa, and OpenSearch all inherit (Elastic docs explicitly cite the Cormack 2009 paper and use `rank_constant = 60` by default). Worth pinning the citation in DESIGN.md so a future maintainer doesn't relitigate it. Important point that DESIGN.md does *not* surface: RRF is rank-based, so the value of k matters less than the *quality* of the two input rankings — i.e. fixing the embedding-truncation problem above is far higher leverage than tuning k. Tell the benchmark seat to A/B chunking and the embedding model first, k second.
  *Recommended action:* edit DESIGN.md § Search → Tutorials to add Cormack 2009 citation and a brief note on why k tuning is low priority relative to ranker quality.

- **[severity: medium]** **[scope: design]**
  Chunking strategy ("H2 with H3-split on overflow") will struggle on real Godot RST. Spot-checking two pages: `gdscript_basics.rst` has 1 H1, ~25–30 H2, ~40+ H3 — a 25-section reference page where H2 boundaries produce well-sized chunks. `2d_lights_and_shadows.rst` has 1 H1, 8 H2, only 2 H3 — and one of those H2 sections ("Setting up shadows") is documented as "particularly substantial, spanning property explanations through multiple shadow techniques and filtering options" with no H3 to split on. The H3-split overflow rule won't help here because there are no H3s. The current chunker has no defined behavior for "H2 chunk > hard cap, no H3 boundaries available" — best guess from the design is "chunk truncated or single oversized chunk emitted," neither acceptable. Add a third tier: when H2 > hard cap and no H3 is present, split on paragraph boundaries (double-newline) preserving code blocks intact. Also: be explicit about how to handle pages with no H2 at all (some are 1 H1 + flat prose); current rule "chunk by H2" implies a single chunk containing the whole page, which will trivially blow the hard cap.
  *Recommended action:* edit DESIGN.md § Ingestion pipeline step 6 with explicit fallback hierarchy: H2 → H3 → paragraph → token-window-with-overlap.

- **[severity: low]** **[scope: design]**
  DESIGN.md § Search → Tutorials: "Embedding model loads lazily on first tutorial search (~1s cold). Subsequent queries fast." Combined with the cold-start of `@xenova/transformers` ONNX session setup, real cold latency is probably 2–4 s on first call, not 1 s. For an agent invocation this is borderline noticeable. Two cheap options: (a) preload the model in a background task after `process.exit(0)`-safe init completes — fire-and-forget `await import('./embedder.js').then(e => e.warm())`, no latch wait; (b) only preload when `GODOT_DOCS_EAGER_INIT` is set (mirroring the LSP eager-init env var). Either way, document the choice. Today's design implicitly accepts the cold-start hit on the first tutorial query, which is fine if benchmarks show it stays under MCP client timeouts but worth confirming in the benchmark or research items.
  *Recommended action:* edit DESIGN.md § Search → Tutorials with cold-start mitigation; consider adding `GODOT_DOCS_EAGER_INIT` env var alongside `GODOT_LSP_EAGER_INIT`.

- **[severity: low]** **[scope: design]**
  DESIGN.md § Storage: "Single-row `meta` table with `godot_version`, `godot_docs_branch`, `schema_version`, `indexed_at`, `class_count`, `tutorial_count`, `ingest_warnings`." Missing fields that downstream tools will want: (a) `embedding_model_id` (sha256 or HF revision pinned at ingest time — needed to invalidate cache if the model changes), (b) `ingestion_source_sha` (commit SHA of the godot + godot-docs tarballs, derivable from the Git Trees API or `git/refs` lookup — needed for "is this cache up to date?" and for reproducible-build claims), (c) `ingestion_duration_ms` (cheap, useful for benchmarking). Mention #19 (`godot_docs_info`) currently mirrors the meta table fields exactly, so adding fields there is a single edit in two places.
  *Recommended action:* edit DESIGN.md § Storage and #19 with the three extra fields.

- **[severity: low]** **[scope: design]**
  DESIGN.md § Version resolution step 4: GitHub Tags API unauthenticated rate limit is 60/hr. The 1-hour TTL means a worst-case-busy CI fleet of 60+ concurrent ingestions in an hour will start 403-ing. For a CI build (where `GODOT_DOCS_VERSION=latest` is unlikely — CI should pin a specific version), this is fine. For a self-hosted MCP behind a corporate NAT shared by many developers, it's a hazard. Two low-cost mitigations: (a) document that `GODOT_TOKEN` or `GITHUB_TOKEN`-style env var bumps the limit to 5000/hr; (b) the `latest-resolution.json` cache could persist across runs (it already does — confirm path is *not* in a tmpdir). One sentence in DESIGN.md.
  *Recommended action:* edit DESIGN.md § Version resolution with a note on the 60/hr ceiling and a `GITHUB_TOKEN` opt-in.

## Research items addressed

### #38 — Schema design (Drizzle 1.0 + SQLite extensions)

**Answer (partial, sufficient to decide):** Drizzle ORM is not 1.0 stable as of May 2025. Latest stable is **0.45.2** (March 2025); latest 1.0 pre-release is **v1.0.0-rc.3** (May 2025). The official Drizzle docs at `orm.drizzle.team/docs/extensions/sqlite` state plainly: *"Currently, there are no SQLite extensions natively supported by Drizzle. Once those are added, we will have them here!"* — so neither FTS5 virtual tables nor sqlite-vec are supported by the query builder regardless of which version you adopt. In practice this means Drizzle's only role in this project is type-safe access to the normal `classes`, `members`, and `meta` tables; everything touching `members_fts`, `tutorials_fts`, and `vec_tutorials` will be `db.run(sql\`...\`)` raw SQL with bound parameters either way. Given the small surface, the fallback plan in DESIGN.md ("if Drizzle 1.0 slips, use better-sqlite3 directly") is approximately the same amount of code as going through Drizzle for five tables, and avoids gating ingestion on Drizzle's release schedule. Recommend: pick one in DESIGN.md with a tripwire date ("if 1.0 not GA by 2025-XX-XX, default to better-sqlite3"). Sources: https://github.com/drizzle-team/drizzle-orm/releases (release list), https://orm.drizzle.team/docs/extensions/sqlite (extension support statement), https://orm.drizzle.team/ (1.0-beta language in landing page banner). Remaining: confirm the exact better-sqlite3 + sqlite-vec loading pattern works on Windows prebuilds (npm tarball sparse on docs at this time; the supply-chain seat will own this).

### #39 — FTS5 tokenizer choice and BM25 weights

**Answer (partial, with concrete starting point):** SQLite FTS5's `bm25()` is positional with default-1.0 weights for missing trailing args (per https://www.sqlite.org/fts5.html). DESIGN.md's "name fields ~3x description" maps to:

- `classes_fts(name, brief)` → `ORDER BY bm25(classes_fts, 3.0, 1.0)`
- `members_fts(name, signature, description)` → `ORDER BY bm25(members_fts, 3.0, 2.0, 1.0)` (signature in between because it contains identifiers; tune in #32)
- `tutorials_fts(title, heading_path, content)` → `ORDER BY bm25(tutorials_fts, 3.0, 2.0, 1.0)`

DESIGN.md should commit to those as v1 defaults so the benchmark has a baseline to A/B against rather than a "verify or adjust" with nothing to verify. Tokenizer: default is `unicode61`, which lowercases + strips diacritics — fine for prose. For Godot specifically, two tokenizer options are worth considering: `unicode61 "tokenchars=_"` (so `add_child` stays as one token, helpful for member-name lookups) and `trigram` (substring matching — agents querying `Anim` would match `AnimationPlayer`). Trigram is more expensive at index time but the corpus is small (a few thousand class/member rows). Recommend starting with `unicode61 "tokenchars=_"` for class/member FTS and default `unicode61` for tutorial FTS; reconsider trigram if benchmark #32 shows recall problems on partial-name queries. Sources: https://www.sqlite.org/fts5.html (bm25 + tokenizer reference). Remaining: actual weight tuning is by design pushed to the benchmark — partial answer is intentional per the design's "decide after seeing real query patterns" framing.
