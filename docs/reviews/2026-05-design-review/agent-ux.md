# Agent UX / tool-surface review

Lens: I'm reading the 16 `godot_*` tools as a routing model would see them at call time. The questions I'm asking: which pairs will collide on a one-shot routing decision? Are the parameter shapes consistent enough that an agent can transfer a pattern from one tool to the next? Where will an LSP-trained model's priors be miscalibrated against this surface?

## Findings

- **[severity: high]** **[scope: design]** _Canonical tool descriptions are not actually drafted anywhere, yet they are the primary routing signal._ DESIGN.md § "Tool descriptions" (lines 100–112) commits to two strong properties — "first sentence is the primary routing signal" and three named "disambiguation pairs" — but neither the design doc nor any of issues #14–#29 contains the actual description strings that will ship in the JSON schemas. What lives in each issue body is a paraphrase of the design's bullet, not a routable string. Benchmark #30 (tool-routing accuracy) literally cannot run until those strings exist; today it would benchmark whatever each implementer ad-libs. _Recommended action:_ open a new issue, `Draft canonical tool descriptions for all 16 godot_* tools`, blocking #30 and the docs/LSP tool epics (#7, #9, #10). Include the disambiguation pairs from § Tool descriptions as required test cases. The issue's deliverable is a single source file (e.g. `src/tools/descriptions.ts`) so the strings stay aligned across PRs rather than landing piecemeal in seven separate sub-issues.

- **[severity: high]** **[scope: design]** _The five docs-lookup tools form a dense cluster (`godot_search_api`, `godot_get_class`, `godot_get_member`, `godot_search_tutorials`, `godot_get_tutorial`) that the design's three disambiguation pairs only partially cover._ DESIGN.md lines 107–111 lists three pairs, but the implied graph has six adjacent pairs that all need disambiguation, and the most agent-ambiguous one is missing: `godot_get_class` vs `godot_get_member`-with-no-`kind`, since both return "a thing about a class." A second missing pair is `godot_search_tutorials` vs `godot_get_tutorial` — a model that doesn't notice the "by path returned from" handoff in #18's first sentence may reach for `godot_get_tutorial` with a guessed slug and fail. _Recommended action:_ expand DESIGN.md § Tool descriptions to list six disambiguation pairs (search_api↔search_tutorials, search_api↔get_class, get_class↔get_member, search_tutorials↔get_tutorial, get_class↔docs_info, search_api↔docs_info) and add to each one the disambiguating phrase the routing description must contain. This is the input to the description-drafting issue above.

- **[severity: high]** **[scope: design]** _1-based positions on the wire will mis-route a non-trivial fraction of LSP-trained models' first calls._ DESIGN.md line 426 picks 1-based with the rationale "matches editor convention." But the LSP specification and every prior MCP wrapper a routing model is likely to have seen — `cclsp`, `lsp-mcp`, vscode-jsonrpc samples, the LSP spec on microsoft.github.io — is 0-based at every layer the model has been exposed to. "Editor convention" is the convention of _humans reading column numbers in a status bar_, not the convention of _tool-calling models_. An agent that pastes a position from the LSP spec, a stack trace, or another MCP tool will be off-by-one. The mitigation in the design is "documented in tool descriptions" (line 426) — but that mitigation only works if the first sentence carries it, which competes with the routing signal. _Recommended action:_ either (a) flip to 0-based and note in tool descriptions that positions match LSP convention, or (b) keep 1-based but require every position-taking tool's description to end with the literal phrase "positions are 1-based (line 1 = first line, column 1 = first column)" as a parameter doc, not a tool description sentence, so it doesn't crowd out routing. Update DESIGN.md § Tool-specific behavior with whichever choice and a one-line rationale that addresses model priors, not just human ergonomics.

- **[severity: medium]** **[scope: design]** _The "zero results → empty array, not error" rule is called out as universal in the LSP epic #9 but selectively highlighted in sub-issues (only #20 and #26 mention it)._ DESIGN.md lines 428–433 lists the cases — "Multiple definitions: array," "Zero results: empty array," "signature*help out of context: empty," "document_symbols large file: cap + truncated flag," "hover: truncate at 5000." These are written as a flat list at the same level, which obscures that "empty, not error" is meant to apply to every read tool, not just to the two that say so. An implementer wiring up `godot_workspace_symbols` (#24) or `godot_document_symbols` (#23) is reasonably likely to bubble an LSP no-result into an MCP error because the spec lets them. \_Recommended action:* edit DESIGN.md § Tool-specific behavior to promote "Zero results: empty array (not error)" to a single top-level rule that explicitly applies to all seven read-only LSP tools, and add a one-line note to issues #21, #22, #23, #24 stating "applies the cross-cutting zero-result rule from epic #9."

- **[severity: medium]** **[scope: issue:#14]** _`godot_search_api` returning an MCP error for empty-query-no-filters is the wrong shape for an agent._ DESIGN.md line 271 and #14's body specify an error string `"Provide at least a query, inherits_from filter, or category filter."` An MCP error response forces the routing model to react to a failure rather than recover with the next call, and the message is non-actionable on its own — the agent now has to re-read the schema. From a routing-recovery standpoint, the cheaper response is a successful empty `results: []` with a top-level `hint` field carrying the same message. The agent sees a structured "you need to refine your query" instead of an error, and the trace stays clean for benchmark #2. _Recommended action:_ update #14 and DESIGN.md § Search → Class reference to return `{ results: [], hint: "..." }` (HTTP-200 equivalent) for the empty/no-filter case, reserving MCP errors for invalid arguments (negative limit, bad filter value).

- **[severity: medium]** **[scope: issue:#16]** _`godot_get_member` returning an array when `kind` is unspecified clashes with the singular noun in the tool name and with the singular-record shape of its peer `godot_get_class`._ The name "get*member" reads as "fetch one record"; an agent that expects `result.signature` will instead get `result[0].signature` whenever `kind` is omitted, and will silently work in tests where there's no collision then break on a class where a method and constant share a name. The justification in DESIGN.md line 259 ("cross-kind member name uniqueness not enforced") is a schema decision, not a UX one. *Recommended action:_ either (a) rename to `godot_find_member` to match the array semantics, mirroring the LSP `find_\*`family, or (b) keep the name and make`kind`either required or default to`method`(most common case), so the response shape is always a single record. Pick one and update DESIGN.md § Tool surface line 77–78 and #16. My recommendation is (a) —`find_member`slots naturally next to`find_definition`/`find_references` and signals to the routing model that the singular response is not guaranteed.

- **[severity: medium]** **[scope: design]** _Parameter naming for position-taking tools is inconsistent between point tools and range tools._ The point tools (`godot_find_definition`, `godot_find_references`, `godot_hover`, `godot_signature_help`, `godot_preview_rename`) take `line`/`character`. The range tool (`godot_code_actions`, #28) takes `start_line` / `start_character` / `end_line` / `end_character`. These are reasonable individually but read as a parameter explosion to a routing model. A range with `{line, character, end_line, end_character}` (omitting the `start_` prefix when only the end is qualified) reads as a natural extension; or, alternatively, a `range: {start: {line, character}, end: {line, character}}` nested object mirrors LSP's own shape and is what an LSP-trained model expects. _Recommended action:_ update DESIGN.md § Tool-specific behavior to declare a single canonical position/range parameter shape and apply it to all position-taking tools. My preference is the nested LSP-style `range` object because it round-trips with anything the agent might have already pasted from LSP docs.

- **[severity: medium]** **[scope: issue:#10]** _The advisory-write response shape can produce a non-unique `before` string that `str_replace` will reject._ DESIGN.md lines 396–420 and #10's canonical JSON show `{line, before, after}` triples. But `str_replace` in Claude Code's `Edit` tool requires `old_string` to be unique in the file; if the `before` line is `    return null` and the function contains five lines of `    return null`, the agent's apply step fails. The MCP knows the line number but doesn't widen the `before` text. The design acknowledges that "if the file content shifts between MCP compute and agent apply, `str_replace` fails cleanly" (line 422) but doesn't address the static non-uniqueness case. _Recommended action:_ update #10's canonical shape to require the MCP to widen `before`/`after` strings until unique (e.g., include preceding/following lines as anchor context, or include leading indentation + the next non-blank line) and add a sub-rule to DESIGN.md § Write operations: advisory pattern. Alternatively, return both a single-line `before` and an `anchor_context` field so the agent can pick the granularity its edit tool wants. Either way, the burden of producing a uniquely matchable string belongs on the MCP, not the agent.

- **[severity: medium]** **[scope: issue:#22]** _Hover truncation at 5000 chars can cut a markdown fence mid-block and corrupt the agent's rendering or downstream regex._ DESIGN.md line 430 and #22 specify "Truncate at 5000 chars" with no smart-truncation. Godot's class reference is long enough that 5000 chars will land mid-snippet often; an unbalanced ```block will affect both how Claude Code renders the assistant message and how subsequent tool calls parse the hover output. *Recommended action:* update #22 to specify "truncate at 5000 chars, but extend forward to the next fence boundary if currently inside a fenced block, or trim back to the previous boundary if extension would exceed a hard cap (say 6000)." Add a`truncated: true`flag to the response when truncation happens, mirroring the`document_symbols` convention. Update DESIGN.md § Tool-specific behavior to note hover truncation must be markdown-aware.

- **[severity: medium]** **[scope: issue:#24]** _`workspace_symbols` query semantics — "substring, case-insensitive" — are weaker than what an LSP-trained agent expects._ DESIGN.md line 431 and #24 say "Substring, case-insensitive. Passed through to LSP server." Most LSP servers the routing model has seen support fuzzy/CamelCase matching (e.g., `gNbN` matching `getNodeByName`), and the LSP spec leaves the matching algorithm server-defined. An agent that fires `workspace_symbols` with a fuzzy query and gets nothing back may give up on the tool rather than retry with a substring. _Recommended action:_ update #24's behavior to state explicitly: "Matching is substring, case-insensitive — Godot's LSP does not implement fuzzy matching. Agents should issue progressively shorter substrings, not fuzzy abbreviations." Fold the explicit caveat into the tool description's parameter doc so it shows up at routing time.

- **[severity: low]** **[scope: design]** _The advisory-write epic #10 has `rename` baked into the canonical response shape, but two of its three child tools are not renames._ The JSON example in DESIGN.md lines 398–420 and #10 has a top-level `rename: {from, to}` block. For `godot_code_actions` (#28) and `godot_preview_code_action` (#29), that field is either absent or carries a different semantic (e.g. "extract function" has no `from`/`to`). #29 says "same response shape as godot*preview_rename" without saying what `rename` becomes. \_Recommended action:* update DESIGN.md § Write operations: advisory pattern to generalize the top-level field — e.g. `action: {kind: "rename" | "code_action", title: "...", details?: {...}}` — and update #10's acceptance bullet "Response shape matches the canonical JSON above" to reference the generalized shape. Keep the `edits` + `summary` portions identical across all three tools.

- **[severity: low]** **[scope: issue:#19]** _`godot_docs_info` is the agent's only surface for understanding what doc surface they're working against, but its description doesn't signal that to the routing model._ The current first sentence from DESIGN.md line 80 ("Get information about the documentation currently loaded") and #19's body don't tell the model when to call it. A model never calls it preemptively. The high-value moment is _after_ `godot_get_class` returns "class not found" — the agent wants to know whether the loaded docs version is the one they expect before guessing a typo. _Recommended action:_ in the canonical description (see first finding), specify the first sentence as something like "Report which Godot docs version is loaded and how many classes/tutorials are indexed; call this when class lookups unexpectedly fail or when reasoning about API availability across versions." Update DESIGN.md § Tool surface line 80 to reflect the routing-signal wording, even if shortened.

- **[severity: low]** **[scope: design]** _No tool description tells the routing model that hover, find-definition, etc. are about user code, while `godot_search_api` / `godot_get_class` are about engine API._ The user-code-vs-engine-API split is the single most important routing distinction for an LSP-trained agent who, by default, will reach for `find_definition` on `Node.add_child` and get either nothing back, a synthetic URI, or an engine-internal path (DESIGN.md line 619 acknowledges this is undetermined). The disambiguation pair "search*api vs search_tutorials" doesn't capture the more fundamental "search_api vs find_definition." \_Recommended action:* add a disambiguation pair to DESIGN.md § Tool descriptions covering user-code-vs-engine-API: `godot_find_definition` is for user GDScript symbols, `godot_get_class` / `godot_get_member` for built-in Godot types. Whichever of #34's research outcomes lands, the first-sentence guidance should steer routing away from LSP for built-in symbol lookups even if Godot's LSP does return something useful for them.

## Open questions

- Should `godot_signature_help` exist as a standalone tool at all? Routing models call it almost exclusively during code completion, but this MCP has no completion tool. In an agent context, the agent reads the function signature via `godot_hover` on the name, not via signature_help on a position inside a call. I'd consider dropping it, but that's outside my lane and intersects with #35 (capability advertisement).
- For `godot_get_member` with `kind` unspecified, should the result be `array` always, or `array` only when collisions exist and `object` when there's exactly one hit? The latter is more ergonomic but means the agent has to type-check; the former is consistent but feels wasteful for the common case. Either way the tool name should be reconsidered (see finding above).
- The advisory-write epic #10 says `str_replace` is the agent's apply mechanism, but `str_replace` is a Claude-Code-specific naming. Other MCP clients have different edit tool names. Is the response shape contractually tied to Claude Code, or should the field names be more neutral (e.g. `find` / `replace` rather than `before` / `after`)?

## Out of scope

- Whether the latch pattern handles initialization timing correctly for the routing tests — that's systems architect territory.
- Whether benchmark #30 is methodologically sound (sample size, scoring, holdout) — that's the benchmark seat's call. I'm only flagging the _input precondition_ (canonical descriptions exist) that affects whether it can run at all.
- The choice of FTS5 weights and whether substring search on workspace_symbols competes with FTS5 on classes — that's docs/retrieval specialist territory.
- Native dep choice for embedding model — supply-chain seat.
