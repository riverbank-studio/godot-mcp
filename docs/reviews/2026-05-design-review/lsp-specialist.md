# LSP / Godot integration review

Audit of the LSP subsystem in `docs/DESIGN.md` against prior art (cclsp, lsp-mcp, opencode-godot-lsp) and known Godot 4.x LSP behavior. All web evidence is cited; "needs real Godot" is flagged where desk research can't fully resolve.

## Findings

- **[severity: high]** **[scope: design § Connection lifecycle / Process management; issues #8, #36]** **The "Godot LSP is single-client; attaching kicks IDE clients" premise is wrong.** Godot's GDScript LSP accepts **up to `LSP_MAX_CLIENTS = 8` concurrent clients** (see `modules/gdscript/language_server/gdscript_language_protocol.h`; the new-client handler accepts and pushes into a `HashMap<int, Ref<LSPeer>>` rather than evicting an existing peer — confirmed in [gdscript_language_protocol.cpp WebFetch above and godot-proposals#11056: "we are allowing multiple connections to the server, which the specification forbids"](https://github.com/godotengine/godot-proposals/issues/11056)). The design uses this incorrect assumption to justify "each MCP session has its own headless Godot" and to defer "attach to user's existing editor" to Future Work. In reality: (a) attaching to the user's running editor is a perfectly viable v1 path, not a future-work item; (b) running headless still works but the _reason_ given is incorrect — the actual reasons are (i) the user may not have the editor open, (ii) one Godot process per project is the only way to query a project the user isn't currently editing, and (iii) avoiding the broken-disconnect bug in [godot#75849](https://github.com/godotengine/godot/issues/75849) where peers don't get cleaned up and the 8-slot pool fills. _Recommended action:_ edit DESIGN.md § Process management → "Why isolated processes" to correct the premise (cite the 8-client limit and the disconnect-leak bug as the real reasons, not single-client kicking); reword the corresponding sentence in #8; resolve #36 with this finding (it is NOT single-client) and reframe the Future-Work bullet "Attach to user's existing Godot editor LSP" as a near-term v1.1 option rather than waiting on a single-client check.

- **[severity: high]** **[scope: design § Tool surface; issue #10]** **`godot_code_actions` and `godot_preview_code_action` will not work in v1 because Godot's LSP does not advertise `codeActionProvider`.** [godot-proposals#14307](https://github.com/godotengine/godot-proposals/issues/14307) is an _open_ feature request (Feb 2026) to add `textDocument/codeAction` for method-stub generation; community reports note `"codeActionProvider": null` in Godot's initialize response (search result above). The design ships these two tools as if they'll work. _Recommended action:_ edit DESIGN.md § Tool surface to either (a) drop both code-action tools from v1 and move to Future Work, or (b) ship them with an explicit "currently always returns `[]` because Godot's LSP does not implement codeAction; will activate once godot-proposals#14307 lands" caveat in the tool description. Update issue #10 acceptance criteria accordingly. The "10 LSP tools" headline count drops to 8 unless you keep them as no-op stubs.

- **[severity: high]** **[scope: design § Document tracking, Connection lifecycle, Concurrency; issue #8]** **Full sync (`textDocumentSync.change: 1`) + "auto-resync before every query" re-sends the entire file on every single LSP request, which Godot itself flagged as a known performance bottleneck.** [godot#87410 "Very slow LSP diagnostics"](https://github.com/godotengine/godot/issues/87410) (closed 4.3 milestone) explicitly identifies "every time you hit a key, the entire text of the document is sent" + "diagnostics on every change" as the root cause of slowness on large files, and the fix tracked there is to move Godot's _server_ to incremental sync — but the server currently only supports full (`gdscript_language_protocol.cpp` says "We only support TextDocumentSyncKind::Full"). For the MCP, the multiplier is worse than the IDE case: an agent doing `find_definition` → `find_references` → `hover` on a 5000-line script triggers 3× full-file resync. cclsp's document manager (WebFetch above) also uses full-text content changes (`contentChanges: [{ text }]`) but does **not** force a pre-query disk re-read; it only sends `didChange` when the caller hands it new text. The MCP's auto-resync-on-every-query is strictly worse than cclsp's approach. _Recommended action:_ edit DESIGN.md § Document tracking → narrow auto-resync to "before queries on files whose mtime changed since last sync" (cheap `fs.stat` check) rather than "re-read content and compare." Add an explicit benchmark item to research-list #37 measuring a `find_references` round-trip on a 5000-line script with and without auto-resync, and gate the auto-resync default on the result. Note in #8 that `vscode-jsonrpc` has no built-in throttling.

- **[severity: high]** **[scope: design § Diagnostics; issue #8]** **The 2-second `publishDiagnostics` await after `didChange` is unrealistic for Godot's cold-parse path.** The same [godot#87410](https://github.com/godotengine/godot/issues/87410) report describes diagnostics being "noticeably slow on large files" even with threading enabled, and `opencode-godot-lsp`'s own README warns "Startup Time: first connection takes 10-20 seconds." For a freshly-spawned headless Godot the first `didChange` on a file triggers a full project parse, not just that file. 2s will routinely time out. _Recommended action:_ edit DESIGN.md § Diagnostics to differentiate first-touch from steady-state: e.g., 10s timeout on the first diagnostic-await per file URI in a session, 2s on subsequent awaits, surfaced as `partial: true` with cached diagnostics rather than an error on timeout. Either that, or remove the deterministic-await contract entirely and document `godot_get_diagnostics` as "best-effort — returns the latest pushed diagnostics; may be stale if the server is still parsing." Update #8 to reflect whichever policy is chosen.

- **[severity: high]** **[scope: design § Write operations; issue #10]** **The `{file, line, before, after}` per-line edit shape cannot represent two non-overlapping edits on the same line, which is normal for renames.** Example: `var x = old_name(old_name(1))` — an LSP `WorkspaceEdit` for a rename of `old_name` returns two TextEdits with disjoint ranges on the same line. The DESIGN-doc shape collapses them into one `(line, before, after)` triple, and any naïve implementation will either (a) ship two triples with the same `before` text, breaking `str_replace`'s uniqueness requirement, or (b) ship the line twice with overlapping `after` text, both wrong. cclsp avoids this by passing through LSP `WorkspaceEdit` ranges directly. _Recommended action:_ edit DESIGN.md § Write operations to either (a) merge per-line same-line edits into a single triple whose `after` reflects all edits applied (and the schema's job becomes "compute the post-edit line"), or (b) switch the response shape to LSP-native `{file, range, newText}` triples and let the agent do the substitution itself. Update issue #10 acceptance criteria with a same-line test case ("rename `old_name` in `old_name(old_name(1))` produces exactly one canonical edit per occurrence"). Note: this same shape is consumed by `godot_preview_rename`, so the fix has to propagate to both tools.

- **[severity: medium]** **[scope: design § Concurrency; issue #8]** **The single-flight 10-second queue lets a slow `find_references` head-of-line-block `hover` / `signature_help` for a full 10s.** cclsp's `operations.ts` (WebFetch above) uses a `30000`-ms default and lets per-method adapters override (e.g., Pyright extends references to 60s); it doesn't serialize globally. In an agent flow, `signature_help` and `hover` are interactive-feel — being stuck behind a workspace-symbol query for ~5s degrades the experience materially. _Recommended action:_ edit DESIGN.md § Concurrency to either (a) allow up to N concurrent LSP requests (JSON-RPC IDs are already correlated, so the only obstacle is Godot's threading bugs — see [godot#79525](https://github.com/godotengine/godot/issues/79525), but that's been fixed since 4.2); or (b) keep single-flight but separate the queue into two priorities: `hover`/`signatureHelp`/`documentSymbol` jump ahead of `references`/`workspaceSymbol`/`rename`. Either way bump the default timeout from 10s → 30s to match cclsp's safe baseline, and let a future per-server adapter (#13) override per method. Update #8.

- **[severity: medium]** **[scope: design § Process management; issue #8]** **"Cap 3 spawn cycles per session, then mark LSP permanently unavailable" leaves the user stranded with no in-band recovery — only an MCP restart unblocks them.** This is acceptable for first cut, but the failure-mode communication is silent: an agent that ran healthy tools for an hour suddenly gets "LSP unavailable" with no remediation hint. _Recommended action:_ edit DESIGN.md § Process management → death detection: when the third spawn cycle fails, surface in the tool error a (a) `recovery_hint: "MCP restart required"` field, and (b) document a `godot_lsp_restart` admin tool (even if not in v1 — see also benchmarking concerns about being able to reset state between runs). Optionally add a `reset_after_minutes` env var to auto-reset the spawn-cycle counter so a transient flake doesn't poison the rest of the session. Update #8.

- **[severity: medium]** **[scope: design § Implementation phases; issue #12]** **Deferring symbol-based fallback resolution (cclsp `findSymbolsByName`) to v1.1 will hurt agent usability in v1.** cclsp shipped symbol-based resolution precisely because agents give imprecise positions — they say "the `process_input` function" and don't know `(line=47, character=4)`. The MCP equivalent failure is: agent asks `godot_find_definition` at `(file=X, line=47, char=4)` based on its last grep, but the file shifted by two lines from an unrelated edit; LSP returns nothing; agent gives up or re-greps. cclsp's `findSymbolsByName` is implemented in ~80 lines (per the operations.ts inspection above — flattens documentSymbols, matches by exact-or-includes, falls back across symbol kinds). It is not a v1.1-sized investment. _Recommended action:_ either (a) pull #12 into v1 scope with a 1-issue acceptance "given `(file, symbol_name)` without position, return matches resolved via `documentSymbol`," or (b) at minimum, accept `symbol_name` as an alternative parameter to `(line, character)` on `godot_find_definition`/`godot_find_references`/`godot_hover` in v1 even if the implementation in v1 just calls `workspace_symbol` and returns the first match. Reframe issue #12 from "follow-up" to "v1 part 2."

- **[severity: medium]** **[scope: issue #13]** **The per-server adapter slot (#13) is described as "initially empty," but there are at least four concrete Godot quirks that will need adapter workarounds and naming them now prevents the abstraction from being shaped wrong.** From this review's research:
  1. `workspace/symbol` returns empty results — [godot-vscode-plugin#989](https://github.com/godotengine/godot-vscode-plugin/issues/989) — so `godot_workspace_symbols` will need a fallback (e.g., union of `documentSymbols` over all tracked-open `.gd` files).
  2. Hover returns format that breaks in non-VSCode clients — [godot#87192](https://github.com/godotengine/godot/issues/87192) — so the adapter may need to normalize `MarkedString`/`MarkupContent` shape to ensure markdown.
  3. Built-in symbol definitions (`Node.add_child`) probably return either nothing or a synthetic URI not on disk — see research #34 — adapter needs to special-case "if URI is unreadable, redirect to docs subsystem `godot_get_member`."
  4. `find_references` on autoload globals (`MyGameManager.do_thing()` where `MyGameManager` is an autoload) is highly suspect given Godot's broader autoload identifier-resolution bugs ([godot#3156](https://github.com/godotengine/godot/issues/3156), [godot#32759](https://github.com/godotengine/godot/issues/32759), [godot#71296](https://github.com/godotengine/godot/issues/71296)). The adapter may need to scan `project.godot` for `[autoload]` entries and union with text-grep results.
     _Recommended action:_ edit issue #13 body to enumerate these four quirks as "expected initial inhabitants of the adapter" so the abstraction is designed for them, even if implementation is deferred. Also reference cclsp's Pyright adapter (per-method timeout overrides) and Vue adapter (non-standard protocol handling, custom request fallback) as the two shape templates — Godot is more like Pyright (timeouts + per-method polish) than like Vue (custom protocol).

- **[severity: medium]** **[scope: design § Document tracking; research item #37]** **Auto-resync-on-every-query is probably _necessary_ (not redundant) because Godot's LSP has no file watcher.** Confirmed by inspection of `gdscript_language_protocol.cpp` (WebFetch above): publishDiagnostics is only triggered from `lsp_did_open` and `lsp_did_change`, with no external file-system watcher. Separately, Godot's editor-side "Auto Reload Scripts on External Change" feature is known buggy ([godot#49298](https://github.com/godotengine/godot/issues/49298), [godot#69485](https://github.com/godotengine/godot/issues/69485)) and applies to the editor's _script editor view_, not its LSP server — so the editor's file watcher does not feed back into LSP state. Conclusion: without our `didChange`, an external edit to a `.gd` file (by Claude's `Edit` tool, by another agent, by `git checkout`) will not be observed by the LSP server, and all subsequent queries will be against stale parsed AST. _Recommended action:_ close research item #37 with the answer "Godot's LSP does NOT push diagnostics on external disk changes; auto-resync is required for correctness." Reference the protocol source. Combine with finding #3 above: keep auto-resync, but narrow it to mtime-changed files.

- **[severity: medium]** **[scope: design § Initialization failure semantics, Process management; issue #8]** **The `--editor --headless --lsp-port` spawn line works on current Godot 4.x but has a startup latency hazard not surfaced in the design.** Confirmed against [Godot CLI docs](https://docs.godotengine.org/en/stable/tutorials/editor/command_line_tutorial.html): `--lsp-port` exists and is editor-only, `--editor` is required (it's what enables the LSP feature), `--headless` is compatible. So the design's command is correct. However `opencode-godot-lsp`'s README explicitly warns "first connection takes 10–20 seconds" because the editor performs project import on first run. The DESIGN.md `GODOT_LSP_EAGER_INIT=false` default means the _first agent LSP tool call_ in a session takes 10–20s, which will likely time out at the MCP-client level. _Recommended action:_ edit DESIGN.md § Process management to document the cold-start cost prominently and recommend `GODOT_LSP_EAGER_INIT=true` for interactive agent use; the lazy default optimizes for the wrong axis (saving spawn time when LSP is unused) when the realistic case is "agent will use LSP within 30s of startup, but the first call will time out if lazy." Also: confirm via real-Godot integration testing that the spawn arg order matters — some Godot versions require `--editor` _before_ `--headless`. Update #8 acceptance criteria.

- **[severity: low]** **[scope: design § Tool surface]** **`godot_signature_help`'s "returns empty (not error) when out of context" is unrealistic given Godot's signature-help quirks.** [godot#51617](https://github.com/godotengine/godot/issues/51617) documents that `.new()` returns `GDScript.new()` docs rather than `_init()` of the actual class, and that signature help fails when args span multiple lines. The "empty when out of context" semantics will be wrong: it'll often return _the wrong_ signature confidently. _Recommended action:_ edit DESIGN.md § Tool-specific behavior `signature_help` paragraph to acknowledge the `.new()`/multi-line-args quirks and warn the agent in the tool description that GDScript signature help is unreliable on constructor calls. This is an adapter-pattern (#13) candidate too.

- **[severity: low]** **[scope: design § Process management; research item #33]** **The `XDG_RUNTIME_DIR`-style "no project lockfile" inference is correct, but the consequence is subtler than the DESIGN's "report unavailable for second session."** Confirmed via [godot-proposals#8593](https://github.com/godotengine/godot-proposals/issues/8593): the contention is _port-level_, not project-level — two headless Godots on the same project work fine _if_ they bind different ports. The DESIGN.md port-upward-scan already handles this. _Recommended action:_ close research item #33 with the answer "no project-level lockfile; only port-level contention, which the existing upward port scan handles. Two concurrent MCP sessions on the same project will both succeed." Edit DESIGN.md § Edge cases and known limitations to retract the "Two MCP instances on the same project may fail to spawn" warning — the port scan resolves it. Update #33 with this resolution.

- **[severity: low]** **[scope: design § Configuration]** **`GODOT_LSP_PORT` default of 6005 is fine, but `GODOT_LSP_HOST=127.0.0.1` is the only sane value and exposing it as a knob invites WSL/devcontainer users to set it to `0.0.0.0` thinking it'll help, exposing the LSP to the LAN.** Godot's LSP has no authentication. Recall that an LSP that allows arbitrary script execution effectively over the wire (workspace-edit, code execution via the broader Godot DAP if conflated) is a security risk on a non-loopback bind. _Recommended action:_ edit DESIGN.md § Configuration → either drop `GODOT_LSP_HOST` entirely (hardcode loopback) or add a stern warning "Do not set to non-loopback unless you understand the security implications." Lean toward dropping it; the MCP spawns the Godot child, so the host should always be loopback to the spawning process.

## Research items addressed

- **#33 (concurrent project access).** **Resolved via desk research.** Godot's LSP does not use project-level lockfiles. The only contention point is the TCP port (`--lsp-port`). The DESIGN's existing port-upward-scan already mitigates this — two MCP sessions on the same project will land on ports 6005 and 6006 and both work. Source: [godot-proposals#8593](https://github.com/godotengine/godot-proposals/issues/8593). The DESIGN's "default lean: report unavailable for second session" is over-cautious and can be dropped. **Verification still needs real-Godot integration** for the project-import phase: it's not clear whether two simultaneous import passes on the same `.godot/` cache directory race. Punt that piece to integration testing.

- **#34 (built-in symbol definitions).** **Needs real Godot integration testing — won't fully resolve via desk research.** No definitive documentation found for what Godot returns on `textDocument/definition` for `Node.add_child` or similar built-in symbols. The DeepWiki summary of the GDScript LSP describes a `documentLinkProvider` that "tokenizes the source to find string literals that point to valid files" — implying paths are real disk paths where they exist. For built-ins (engine C++ source not on the user's disk), the most likely behavior is either an empty result or a synthetic `gdscript://` / `godot://` URI; [godot#111400](https://github.com/godotengine/godot/issues/111400) notes that built-ins behave differently from user code in 4.5 ("`Node` displays documentation, but `Dictionary` produces errors"), implying the LSP-side handling is fragile. Recommend integration test: query definition on `Node.add_child`, `Vector2.length`, and `Dictionary.keys`; record what comes back. Plan a `lsp/adapter.ts` fallback that redirects unreadable URIs to the docs subsystem (`godot_get_member`).

- **#35 (capability advertisement).** **Mostly resolved via desk research.** From the protocol source ([gdscript_language_protocol.cpp](https://github.com/godotengine/godot/blob/master/modules/gdscript/language_server/gdscript_language_protocol.cpp)) and the DeepWiki summary, Godot's LSP advertises: `textDocumentSync` = Full (1), `hoverProvider`, `definitionProvider`, `referencesProvider`, `documentSymbolProvider`, `signatureHelpProvider`, `renameProvider`, `completionProvider`, `codeLensProvider`, `documentHighlightProvider`, `foldingRangeProvider`, `documentLinkProvider`, `colorPresentationProvider` — all true. **NOT supported:** `codeActionProvider`, `workspaceSymbolProvider` (de-facto broken even where it claims support — [godot-vscode-plugin#989](https://github.com/godotengine/godot-vscode-plugin/issues/989)), `semanticTokensProvider`, `inlayHintProvider`. _Implication for DESIGN.md:_ both code-action tools and the workspace-symbol tool need caveats (see findings #2 and #9). The capability list in this paragraph should be folded into the DESIGN.md LSP Connection-lifecycle section or into a new "Known LSP server capabilities" subsection. Integration testing should confirm by parsing the actual `initialize` response on a real Godot 4.5/4.6 instance.

- **#36 (single-client behavior).** **Resolved via desk research.** Godot 4.x's LSP supports **up to 8 concurrent clients** (`#define LSP_MAX_CLIENTS 8` in `gdscript_language_protocol.h`; `on_client_connected` adds to a HashMap with no eviction). Sources: WebFetch of the protocol header above; [godot#75849](https://github.com/godotengine/godot/issues/75849) ("LSP and DAP clients aren't properly disconnected... eventually makes it reach the clients limit (8)"). Connecting from the MCP to the user's existing editor LSP does NOT kick the IDE client. However: the disconnect-cleanup bug (peers not removed if `poll()` isn't called) means _transient_ failure of MCP connections will accumulate dead peers and eventually exhaust the 8 slots — though [godot#75849](https://github.com/godotengine/godot/issues/75849) appears to have been fixed in 4.1's PR #75850. _Implication:_ the Future-Work bullet "Attach to user's existing Godot editor LSP as an opt-in mode" is more viable than presented; can be implemented via `GODOT_LSP_ATTACH_TO_PORT` env var without single-client concerns. Spawning-our-own-headless is still preferred as default (because the user may not have the editor open, and to avoid the lingering disconnect leak risk on older Godot 4.0).

- **#37 (file watcher behavior).** **Resolved via desk research.** Godot's LSP does NOT have an internal file system watcher and does NOT push `publishDiagnostics` on external disk changes. From `gdscript_language_protocol.cpp` (WebFetch above): "`publish_diagnostics` is published only reactively... when documents are explicitly opened via `lsp_did_open()` or modified via `lsp_did_change()`, not from external disk changes." Reinforced by the broader cluster of bugs around Godot's "Auto Reload Scripts on External Change" feature ([godot#49298](https://github.com/godotengine/godot/issues/49298), [godot#69485](https://github.com/godotengine/godot/issues/69485)) — the editor itself struggles to observe external edits; LSP definitely doesn't. _Implication:_ auto-resync in the MCP is **required for correctness**, not redundant. Combine with finding #3 to narrow auto-resync from "read content and compare" to "stat mtime, then re-read only if changed." Close #37 with this conclusion.

## Open questions

- Does `--editor --headless --lsp-port` require `--editor` to come _before_ `--headless` in argv? Some Godot versions parsed argv order-sensitively. Worth real-Godot smoke test before locking the spawn-line into code.
- Does Godot 4.5 still have the `--editor --headless` 10–20s cold-start, or has project-import caching improved enough that subsequent spawns on the same project are sub-second? If sub-second on warm cache, the EAGER_INIT recommendation softens.
- What is the actual MCP-client request timeout for Claude Code? If it's ≥30s, the cold-start LSP problem is hidden behind it. If it's much shorter (e.g., 10s), eager init or explicit "warming up" UX is required.
- For `find_references` on autoload globals: does Godot's LSP traverse the autoload registry, or treat the autoload identifier as undeclared (per [godot#3156](https://github.com/godotengine/godot/issues/3156))? Determines whether adapter #13 needs an autoload-aware reference shim.
- Are `vscode-jsonrpc`'s `SocketMessageReader`/`SocketMessageWriter` happy on a Node.js `net.Socket` against Godot's framing? Godot's LSP is supposed to be Content-Length-framed JSON-RPC, but the Godot side has had framing quirks historically.
