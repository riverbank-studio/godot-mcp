# Systems architect review

## Findings

- **[severity: high]** **[scope: design]**
  The "docs init fail → crash, LSP init fail → degrade" asymmetry in § Cross-subsystem independence and § Initialization failure semantics is only defensible for _startup_ failures, but the design quietly extends docs-crashes-the-process semantics to _runtime_ failures as well. § Concurrency model says "On failure, latch rejects and `process.exit(1)`" and § Failure semantics says "Total failure → `process.exit(1)`" — but a cache-miss runtime fetch can happen after the server has been answering tool calls for hours (e.g. user just switched `GODOT_DOCS_VERSION`, or `latest` resolved a new tag after the 1-hour TTL expired). Killing the whole MCP server because a network blip occurred during a _runtime_ docs refetch is dishonest and also breaks the cross-subsystem independence claim: a docs network failure _does_ now break editor tools, contradicting § Cross-subsystem independence ("a docs network failure doesn't break editor tools"). The design needs to distinguish "init-time docs fail = crash" from "runtime docs refetch fail = mark unavailable, surface MCP error, keep server up." _Recommended action:_ edit DESIGN.md § Documentation subsystem → Concurrency model and § Failure semantics to scope `process.exit(1)` to the cold-startup ingestion path only; specify the runtime-refetch failure mode explicitly.

- **[severity: high]** **[scope: design]**
  § LSP subsystem → Process management caps "spawn cycles per session at 3. After that, mark LSP permanently unavailable for the session," with no operator-visible recovery primitive. In an MCP session that lasts a workday this is brutal: a transient port collision or a single OOM during early experimentation locks the LSP off for the rest of the session, and the recovery UX is "tell the user to restart the MCP server" — which on Claude Code means killing the whole transport. The cap exists for a real reason (don't pin loop on a broken config), but it should reset on success (e.g. one good handshake clears the counter), or expose a `godot_lsp_reset` tool, or window the cap (3 cycles per 10 minutes) rather than session-lifetime. As written this primitive trades a rare pathology for a common-case footgun. _Recommended action:_ edit DESIGN.md § LSP subsystem → Process management to specify cap-reset semantics (on successful handshake, or windowed), and either add a manual-reset tool or explicitly document the "restart MCP" remediation in § Edge cases.

- **[severity: high]** **[scope: design]**
  § Document tracking specifies "Auto-resync before queries: Before any LSP query, re-read tracked files from disk; if content differs from last-sent, send `didChange`." This is O(tracked-files × file-size) syscalls + content compare on every single LSP tool call. For the realistic case of an agent in a mid-size project that has touched 30–50 `.gd` files over a long session, every `godot_hover` does dozens of file reads and string diffs. For the 5000-line script case it's a multi-megabyte read+hash per call. The design hand-waves the cost ("Memory cost trivial" appears in Diagnostics; nothing addresses the I/O cost). The honest design is to stat-then-read (compare mtime+size first; only hash/read if it changed), and even then bound it to the file(s) the _current_ tool call references plus any whose mtime changed since last sync. § Research items mentions investigating whether Godot's LSP push covers external edits — until that's answered, the broad-net resync is unjustified. _Recommended action:_ edit DESIGN.md § Document tracking → "Auto-resync before queries" to require an mtime/size short-circuit before content read, and scope the broad resync to "tracked files whose stat changed" rather than all tracked files.

- **[severity: medium]** **[scope: design]**
  The "latch primitive" called out as shared infrastructure in § Shared infrastructure and § Concurrency (LSP) is just `Promise<void>` constructed once and awaited many times — i.e. `const ready = doInit()` and `await ready` at every entry point. § Concurrency model itself describes it as a "one-shot promise." Introducing a named abstraction for this is ceremony, not load-bearing — both subsystems literally duplicate the same shape because there is no shape to share. A named primitive _would_ be load-bearing if it added: resolved/rejected/pending introspection (LSP needs this to advertise "unavailable"), reset-on-recovery (LSP needs this after a successful respawn), or a "drain" semantics for in-flight rejection. The design names the primitive but doesn't actually impose any of those, so what gets built is two identical inlined promises with a shared filename. Either give it real surface area or drop the name. _Recommended action:_ edit DESIGN.md § Shared infrastructure to either (a) drop the "latch pattern" callout and let each subsystem use a plain init promise, or (b) define the actual API (states, reset, in-flight rejection) it must provide so issue #5's unit-test acceptance criteria are meaningful.

- **[severity: medium]** **[scope: design]**
  § Concurrency (docs) — "Cache miss → kick off async fetch + parse; latch starts unresolved. Tool handlers `await` the latch before querying. ... Tool calls block on the latch until ready or until the server dies. This is acceptable because indexing is fast (typically <30s) and well below MCP client timeout limits." Two problems. First, "typically <30s" is asserted, not measured: parsing thousands of XML files + RST chunking + MiniLM embedding of every tutorial chunk in a fresh-cache scenario is plausibly minutes on a cold machine, especially first run when the ONNX runtime also has to warm. Second, MCP clients vary in tool-call timeout (Claude Code defaults to a few minutes; other clients are stricter), and blocking _any_ tool call — even a docs-unrelated one — behind the latch is broader than needed. Only docs tools should await the docs latch; editor and LSP tools should be unaffected. § Cross-subsystem independence implies this, but § Concurrency model's wording ("Tool handlers `await` the latch") is ambiguous. _Recommended action:_ edit DESIGN.md § Concurrency model to explicitly scope the latch await to docs tool handlers only, and replace "typically <30s" with either a measured number or "subject to benchmark; users on cold caches see a one-time delay" plus a documented expected range.

- **[severity: medium]** **[scope: design]**
  § Documentation subsystem → Concurrency uses a lockfile with "check if PID alive AND mtime within 5 minutes." Two real failure modes are unaddressed. (a) **Clock skew / mtime semantics:** the cache dir may live on a filesystem with low mtime resolution or a clock that drifts vs the writing process (containers, VMs, NFS even though it's discouraged elsewhere); a fresh lockfile can appear stale. (b) **Alive-but-hung writer:** the "fast" PID-alive check passes while the actual ingestion has wedged (e.g. ONNX hang, sqlite-vec extension load failure not yet timed out). The 60s wait + reclaim then races with a writer that's about to come back to life. The lock-protocol should at least include a heartbeat (touch lockfile every N seconds) so mtime tracks _liveness of work_, not just creation, and the reclaiming process should hold the lock long enough to be sure (atomic create+exclusive open, not just file existence). _Recommended action:_ edit DESIGN.md § Documentation subsystem → Concurrency to specify heartbeat semantics and use `O_EXCL`-style atomic creation; consider citing better-sqlite3's own DB-level lock as a fallback if the cross-process coordination proves brittle.

- **[severity: medium]** **[scope: design]**
  § Distribution and release — bundled docs DB is built via a `prepublishOnly` script that calls `fetchAndParseVersion`. This means **publishing the npm package** has hard dependencies on (a) GitHub Codeload availability, (b) GitHub API rate limits if `latest` is involved, (c) the godot-docs tarball, and (d) the MiniLM model downloading correctly into the runner. A network blip or a GitHub incident during a release window aborts the publish. The design notes the network fragility for _runtime_ fetches (with retries, threshold, jitter) but doesn't acknowledge that the _publish_ path has the same fragility and worse blast radius — a failed release is more visible than a failed user install. The mitigation is either: cache a known-good DB in the repo (which § Distribution explicitly rejects: "`data/` directory is gitignored") so publish-time fetch is just a refresh, or run the docs build out-of-band (release workflow, separate from publish) and commit the artifact. _Recommended action:_ edit DESIGN.md § Distribution and release → Bundled docs DB to either (a) describe a build-then-commit-then-publish flow rather than `prepublishOnly`, or (b) explicitly accept the publish-time network coupling and document the retry/manual-rerun playbook.

- **[severity: medium]** **[scope: issue:#5]**
  Issue #5's acceptance criterion for the latch — "unit tests covering: resolved-at-start, resolved-async, rejected-async paths" — is the minimum viable test set for an unspecified primitive. If the design clarifies (per the medium-severity finding above) that the latch needs introspection, reset, and in-flight rejection, those properties need acceptance criteria here. Otherwise the primitive is built to pass three trivial tests and then used by LSP for behaviors (mark-unavailable, respawn-and-clear-state) that the test set doesn't exercise. _Recommended action:_ edit issue #5 body to either (a) defer the latch implementation until DESIGN.md pins its API, or (b) add acceptance criteria for the real-world states LSP and docs runtime-refetch actually use.

- **[severity: medium]** **[scope: design]**
  `dispatch.ts` as a dedicated module is borderline. With 16 tools and no nesting (§ Architecture → Module organization: "Tools are flat — no nested categories"), the routing layer is a flat name → function map. The case for a separate file is that it makes "where do I register a new tool?" obvious; the case against is that it adds an import indirection for what could be `Object.assign` in `index.ts` over the per-subsystem tool objects. The design also doesn't say what `dispatch.ts` does _beyond_ routing — no schema validation, no middleware, no instrumentation hooks. If it stays trivial, it's ceremony. If it grows into a real router (validation, OTel span wrapping per tool, error normalization), then it pays rent and that should be described. Right now § Architecture leaves that ambiguous. _Recommended action:_ edit DESIGN.md § Architecture → Module organization to either justify `dispatch.ts` with the cross-cutting responsibilities it owns (validation, OTel wrapping, etc.) or fold it into `index.ts`.

- **[severity: low]** **[scope: design]**
  § Cross-subsystem independence asserts "Editor tools don't have init failures (no startup work)." This is true today and the design uses it to justify not giving editor tools a latch. The risk is YAGNI in reverse: any future editor tool that needs Godot's actual version probed at startup (which `godot_get_version` already arguably does, and any version-gated tool description would), or any tool that wants to cache `list_projects` results, immediately breaks this invariant. The honest framing is "editor tools currently don't have init work; if that changes, they get a latch like the others" rather than asserting a permanent property of the subsystem. Low severity because it's a wording fix, not a structural problem. _Recommended action:_ edit DESIGN.md § Cross-subsystem independence to soften the editor-tools claim from architectural assertion to current-state observation.

- **[severity: low]** **[scope: design]**
  § Distribution and release notes "x64/arm64 Linux/macOS/Windows" prebuilds for the three native deps, but the **runtime** docs-refetch path on a user machine has to download both the godot tarball and the godot-docs tarball, sparse-extract, and run a 5-stage pipeline including ONNX embeddings. Users on locked-down corp networks, behind proxies that strip GitHub Codeload, or on air-gapped machines will silently fall back to "stable bundled DB" (fine) or hit a hard `process.exit(1)` if they set `GODOT_DOCS_VERSION=4.5` (not fine). There's no mention of `HTTPS_PROXY` honoring or "bring-your-own DB" override (e.g. `GODOT_DOCS_DB_PATH` pointing at a pre-built file). For a tool aimed at game devs (frequently on managed corp networks), this is a real adoption blocker. _Recommended action:_ open a new issue scoped to "support proxy / offline / pre-built DB override for docs ingestion."

## Open questions

- What's the actual median + p95 ingestion time on a cold cache? The "typically <30s" claim drives the entire "block tool calls on the latch" UX choice; if it's actually 60-180s on common hardware, the design probably needs a "still indexing" status surface in `godot_docs_info` after all.
- When LSP is marked permanently unavailable after the 3-cycle cap, do _other_ tools — specifically docs and editor — emit any signal to the agent that the LSP capability is gone, or does the agent just keep getting "LSP unavailable" on every LSP call for the rest of the session with no upstream awareness?
- Does the prepublish docs build run with `GODOT_DOCS_VERSION=latest` or pinned? § Distribution says "current Godot stable version at publish time" but doesn't pin it; if it's `latest`, a release one hour after a Godot tag publishes gets a different bundled DB than a release an hour earlier — small, but a reproducibility wart.
- Are `didChange` payloads going to send the _full_ file each time (since § Connection lifecycle pins `textDocumentSync.change: 1` = full sync)? For a 5000-line script edited once per LSP call, that's a 200KB+ message over TCP per call. Worth confirming Godot's LSP doesn't have an incremental mode worth using.

## Out of scope

- **Tool naming and disambiguation pairs in § Tool surface** — naming quality, the rationale for `godot_get_version` vs `godot_get_godot_version`, and the disambiguation prose for `godot_search_api` vs `godot_search_tutorials` belong to the Agent-UX seat.
- **cclsp/lsp-mcp prior-art comparison** in § Related work and § Implementation approach — what to borrow from them and which patterns to skip is the LSP specialist's lane; I only flagged the auto-resync I/O cost on its own merits.
- **Embedding model choice, FTS5 tokenizer/BM25 weights, and chunking caps** (§ Search, § Ingestion pipeline) — retrieval correctness and chunk-size trade-offs belong to the docs/retrieval specialist. I did not evaluate whether MiniLM-L6 is the right model or whether 1500/3000 are the right token caps.
- **Native-dep supply chain risk and the 150MB package size** (§ Distribution and release) — `@xenova/transformers`, `better-sqlite3`, `sqlite-vec` provenance is the supply-chain seat's territory; I only touched bundled-DB build flow because it's an _architectural_ coupling between npm publish and the network.
- **Benchmark acceptance criteria** (§ Testing and benchmarks, Recall@5 ≥ 80%, etc.) — measurability and dataset construction belong to the benchmark seat.
